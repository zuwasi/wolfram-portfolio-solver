01 — What We Built

A local desktop application that simulates theoretical self-managed trading portfolios — with every single mathematical operation executed by the Wolfram Mathematica kernel.

🔄 Forward Simulation

Given an initial deposit, monthly contributions, and a stock ticker — compute portfolio growth over time using real historical market data.

🔍 Reverse Solving

Given a target final amount, solve for the unknown — initial deposit, monthly contribution, or equivalent return rate.

Key distinction: This is a theoretical simulator, not a brokerage tool. It is designed for mathematical modeling, historical backtesting, and solving inverse financial problems.

03 — The Mathematical Model

The portfolio evolves according to a first-order difference equation:

xn+1 = (xn + b) × (1 + pn)
  • xn — portfolio value after period n
  • b — monthly deposit (added at the beginning of each month, before the return is applied)
  • pn — market return in period n, defined as: pn = pricen / pricen-1 − 1
  • x0 = A — initial deposit, present before the first return period
Explicit closed form:
xn = A·∏(1+pk) + Σ b·∏(1+pm)
The product of returns compounds the initial capital; each deposit compounds from its injection point forward.

04 — Why Mathematica Executes ALL the Math

🎯 Arbitrary Precision

Mathematica's N[expr, 20] computes to 20 significant digits — far beyond IEEE 754 double precision (≈15.9 digits). Critical when compounding over hundreds of periods.

🧮 Symbolic → Numeric Pipeline

Solve linear cases symbolically with Solve[], then evaluate numerically. This guarantees exact results for linear unknowns (A, b) — no iteration needed.

⚡ Industrial FindRoot

Mathematica's FindRoot[] uses adaptive step control, automatic Jacobian estimation, and robust convergence — production-grade Newton-Raphson without hand-coding edge cases.

🔗 Unified Engine

One kernel handles everything: product chains, summation series, root-finding, precision control, and output formatting. No library mismatches, no floating-point surprises.

05 — Forward Simulation in Wolfram Language

The actual Wolfram code sent to the kernel:

Module[{x = 300000., ret = {0.05, -0.03, ...}, vals}, vals = {x}; Do[ x = (x + 6000.) * (1 + ret[[i]]); AppendTo[vals, x], {i, 1, Length[ret]} ]; StringRiffle[ ToString[CForm[#]] & /@ N[vals, 20], "," ] ]
  • Do[...] — imperative loop, mirrors the recurrence exactly
  • N[vals, 20] — 20-digit precision for every intermediate value
  • CForm[#] — outputs C-style scientific notation (e.g., 3.21e5) for clean application-layer parsing
  • StringRiffle[..., ","] — comma-separated output, one call returns all portfolio snapshots

07 — The Annual Return Challenge

Solving for equivalent annual return is the hardest problem:

rm = (1 + ry)1/12 − 1 f(ry) = forward_value(A, b, rm(ry), n) − target = 0
⚠ Complexity: The function (1 + ry)1/12 produces complex numbers when ry < −1. Standard Newton-Raphson will diverge into the complex plane.

Our solution: Force real arithmetic in the Wolfram expression:

rm = Sign[1 + ry] * Abs[1 + ry]^(1/12) - 1

Then wrap the result in Re[...] and use CForm to extract only the real part. This keeps FindRoot on the real number line even for extreme initial guesses.

08 — Implications for Investment Decisions

📊 Realistic Goal Setting

By reverse-solving for required monthly deposits, investors can determine exactly how much to save to reach a target — based on actual historical market behavior, not optimistic assumptions.

📉 Risk Awareness

Historical simulations show the actual volatility path — months of negative returns, drawdowns, and recovery periods. This is fundamentally different from a smooth compound growth curve.

🎯 Equivalent Return Clarity

Solving for equivalent annual return from a historical simulation reveals the true CAGR — often very different from the arithmetic mean of monthly returns.

💡 Sensitivity Analysis

Running the same simulation with different start/end dates exposes how timing dramatically affects outcomes — the same strategy can yield +80% or −20% depending on entry point.

09 — What Can Go Wrong: Dangers of the Data

🚨 Survivorship Bias

Testing with tickers that still exist today ignores the companies that went bankrupt. Your backtest only includes winners — inflating apparent returns.

🚨 Past Performance ≠ Future Results

A stock that returned 15% annually for 10 years may return −5% next year. Historical simulation is a descriptive tool, not a predictive one. Markets are non-stationary.

🚨 Adjusted Close Limitations

Yahoo Finance adjusted close accounts for splits and dividends, but may contain errors, revisions, or gaps — especially for less liquid stocks.

🚨 Monthly Aggregation Hides Intra-Month Volatility

A month with +5% return may have had a −15% drawdown mid-month. The model sees only the net result — missing the emotional and margin-call reality of living through it.

10 — More Risks to Consider

🚨 No Taxes, Fees, or Slippage

Real portfolios incur capital gains taxes (15–37%), brokerage commissions, bid-ask spreads, and fund expense ratios. Ignoring these can overstate returns by 2–5% annually.

🚨 Inflation Erosion

A $1M portfolio in 2050 buys far less than $1M today. Without inflation adjustment, nominal gains create a false sense of wealth growth. At 3% inflation, purchasing power halves every 24 years.

🚨 Single-Ticker Concentration Risk

Simulating one stock ignores diversification benefits. Real portfolios hold multiple assets to reduce idiosyncratic risk. A single stock can go to zero; an index rarely does.

🚨 Constant Deposit Assumption

Real life involves job loss, medical emergencies, and lifestyle changes. The model assumes perfectly consistent monthly deposits — a luxury many investors don't have.

11 — Safeguards for Responsible Use

  • Always run multiple scenarios. Test bull markets, bear markets, and sideways markets. Never rely on a single backtest window.
  • Use conservative return assumptions. When using constant-return mode, use rates 2–3% below historical averages to build a safety margin.
  • Cross-validate with index benchmarks. Compare single-stock results against SPY or a total market fund to gauge whether outperformance is real or lucky timing.
  • Apply a Monte Carlo layer. Use the historical return distribution to run 10,000+ randomized simulations and examine the range of outcomes, not just one path.
  • Factor in taxes and fees manually. Reduce simulated returns by your marginal tax rate and expected costs before making decisions.
  • Treat results as scenarios, not forecasts. The output answers "what would have happened" — never "what will happen."
  • Review deposit timing assumptions. Our model adds deposits at month-start. Real DCA may differ. Small timing changes compound over decades.
  • Check solver convergence. Always verify the Diagnostics tab — non-convergent or high-residual results should be discarded.

16 — How to Interpret Results

📈 Forward Simulation Output

The chart shows the actual path your portfolio would have taken — including every dip and recovery. The final value is what you'd have if you started exactly on the start date and contributed every month without fail.

🔄 Reverse Solver Output

When solving for monthly deposit: "You would have needed to deposit $X/month to reach your target." This is backward-looking — the required deposit for future scenarios will differ based on future (unknown) returns.

📊 Equivalent Annual Return

This is the constant annual rate that would produce the same final value. It smooths out all volatility into a single number — useful for comparison, but it hides the pain of drawdowns.

The golden rule: Use forward simulation to understand what happened. Use reverse solving to understand what it would take. Never use either to predict what will happen.
